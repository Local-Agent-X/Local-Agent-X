# AGENTS.md — Invariants & architectural rules

This file encodes the non-negotiable rules for any agent (human or AI) touching
this codebase. Auto-loaded into the system prompt alongside `config/app-manifest.json`.

The JSON manifest answers *what exists*. This file answers *what's allowed*.

---

## Three Lanes (runtime control)

Every change goes through exactly one of:

1. **Runtime state → HTTP API.** Theme, provider, any flip of a live value. Call the existing endpoint. Do NOT edit files to change runtime state.
2. **Self-modification → `config/` directory.** Agent-editable behavior lives here. Edit directly, changes hot-reload.
3. **External sites → `browser` tool.** Never reach out via raw fetch for user-facing web work; drive a real logged-in browser.

If your task doesn't fit a lane, STOP and ask rather than pick the wrong tool.

---

## Escalation ladder (when a lane fails)

1. HTTP API call
2. Direct edit in `config/` or `workspace/`
3. `self_edit` tool — for `src/` surgery, new tools, or missing capabilities

Don't skip. Try the cheapest step first. Only escalate when the one below returned the wrong outcome or the capability doesn't exist.

---

## Directory boundaries

| Path | Who touches | Gitignored | Rules |
|---|---|---|---|
| `src/` | `self_edit` only (protected for direct edit) | no | Core engine. TypeScript. One responsibility per module. |
| `config/` | agent or user | no | Agent-editable behavior (prompt, tools allowlist). Hot-reloads. |
| `public/` | core only via `self_edit` | **no — COMMITTED** | Shipped UI chrome (app.html, tasks.html, and the settings/sidebar panels under public/js/). Don't add user pages here. |
| `workspace/` | agent or user | **YES — local only** | User files, built apps, per-machine customizations. Served at `/apps/<path>`. |
| `packages/arikernel/` | hands off unless fixing a specific Ari bug | no | Vendored security layer. |
| `dist/` | build output — never hand-edit | yes | Regenerated by `npm run build`. |

**Critical distinction — public/ vs workspace/:**
- `public/` is committed. Any file added here ships to every user of Local Agent X on every machine. Only the core engine chrome belongs here.
- `workspace/` is gitignored. Per-machine. Agent-built pages, apps, user files belong here.
- User asks "build me a calendar" → `workspace/apps/calendar/index.html`, pin to sidebar with url `/apps/calendar/` (served by the `/apps/*` static handler). NOT `public/calendar.html`.
- If you're tempted to put something in `public/` that's user-specific or per-install, STOP. It belongs in `workspace/`.

## Code invariants (enforced by review, not tooling)

- **No file over 400 LOC.** If it grows, split into focused modules.
- **No god files.** One responsibility per module.
- **No mentions of Claude, Anthropic, OpenAI, or any AI tool** in commits, PRs, or code comments.
- **Read before edit.** Don't write changes to code you haven't opened.
- **Prefer editing existing files over creating new ones.**
- **Don't add comments, docstrings, or type annotations to code you didn't touch.** Keep the diff focused.
- **No dark mode as default.** Light is default unless the user asks otherwise.
- **No backwards-compat shims for removed features.** If nothing uses it, delete completely.
- **Don't add error handling, fallbacks, or validation for scenarios that can't happen.** Trust internal code and framework guarantees.

---

## Protocol invariants (enforced at build time)

- **Runtime state changes must broadcast.** Any `/api/*` POST that mutates shared state (settings, provider, theme, session) must call `broadcastAll({ type: "settings_changed", ... })` so connected UI tabs reflect the change. Silent writes to `settings.json` are a bug — they desync the UI.
- **Tool results route through `tool-executor.ts`.** Don't bypass it — Ari + policy + approval + RBAC all live there.
- **External content flows through `sanitize.ts` wrappers.** Never paste raw web content into the model context without `wrapExternalContent()`.
- **Agent-facing tools belong in the tool registry (`src/tools/` definitions, wired in `src/tools/registry-build.ts`).** Model can only see what's registered. Register → policy rule in `tool-policies.data.ts` → done.

---

## Security posture

- **Ari Kernel is in-process** (see `src/ari-kernel/`). Every tool call runs through it. Don't add tool execution paths that bypass.
- **Default-deny policy** — new tools need an explicit allow entry in the unified policy table `src/tool-policy/tool-policies.data.ts` (colocated with the tool's kernel class, risk tier, and rate limit).
- **Secrets never in source, logs, or tool results.** Use `{{SECRET_NAME}}` placeholders; server resolves from `secretsStore`.
- **No outbound network from tools without going through SSRF-guarded egress** (`http_request`, `web_fetch`).

Ask the user. Don't guess. Don't over-engineer. Don't predict future requirements.
