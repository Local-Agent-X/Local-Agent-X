# 1B: Funnel all memory writes through one safety gate
Closes: F5
Status: completed

## What changed
A single funnel ÔÇö `src/memory/write-safely.ts` ÔÇö now owns the gate composition
(`normalize Ôćĺ checkMemoryTaint(threshold) Ôćĺ sanitizeForMemory Ôćĺ
redactKnownSecrets Ôćĺ write`). Every memory-write entry point routes through
it. Callers can RAISE threshold but cannot skip the chain.

End-of-turn writes used to bypass the taint check entirely; they now don't.
Auto-extract used to log a warning and continue; it now blocks (with a typed
error the caller logs and returns from). Tool-path behavior is unchanged
externally but the inline gate is no longer reimplemented inside the tool.

A migration audit mode (`LAX_MEMORY_WRITE_AUDIT=1`) converts blocks into
log-only entries at `logs/memory-write-audit.log` so we can soak one session
before flipping enforcement on by default.

## Files changed
- **src/memory/write-safely.ts** ÔÇö new (193 LOC). Exports:
  - `writeMemorySafely({content, source, target, threshold?, mode?})` ÔÇö file writer
  - `appendToDailyLogSafely({...})` ÔÇö keeps `memory.appendDailyLog`'s markDirty + reindex side effects
  - `writeMindFileSafely({...})` ÔÇö same for `memory.writeMemoryFile`
  - `runMemoryGate({...})` ÔÇö gate-only (returns sanitized content) for non-file sinks like `memory.retain`
  - `MemoryWriteBlocked extends Error` ÔÇö typed throw with `{reason, injectionScore, source, target}`
  - `MemoryWriteSource = "tool" | "eot" | "auto-extract" | "sync" | "personality"`
- **src/memory/tools/save.ts** ÔÇö dropped the inline taint chain; `memory_save` dispatches via `writeMindFileSafely` / `runMemoryGate` / `appendToDailyLogSafely`; `memory_update_profile`'s direct `atomicWriteFileSync` calls (final write + `.bak` backup) now route through `writeMemorySafely`. Both tools catch `MemoryWriteBlocked` and surface `BLOCKED: <reason>` as `isError: true`.
- **src/memory/end-of-turn-write.ts** ÔÇö `applyWrite` previously wrote with raw `atomicWriteFileSync` (no taint check at all). Now routes through `writeMemorySafely` with `source: "eot"`; the per-call `MemoryWriteBlocked` is logged and silently skipped (same UX rule as the existing classifier-failure path: end-of-turn passes never surface errors to the user). Also lifted four `await import(...)` calls to static imports (`fs`, `path`, `os`, `./personality.js`).
- **src/memory/auto-extract.ts** ÔÇö IDENTITY.md / USER.md writes now route through `writeMemorySafely`; daily-log appends go through a `safeAppendDaily` wrapper that catches `MemoryWriteBlocked` so the fire-and-forget caller never crashes. The pre-flight `checkMemoryTaint` skip on the raw user/assistant message is kept (cheap, prevents the LLM classifier from even seeing obvious junk), but the load-bearing block is now the per-write gate, not a warn-only side check.
- **src/memory/personality.ts** ÔÇö `ensurePersonalityFiles`'s default-template writes (`writeFileSync`) now go through `writeMemorySafely` with `source: "personality"`. Defaults are static strings, so they always pass; funneling them keeps the audit invariant tight ("every write through the gate, no exceptions").
- **src/memory/manager.ts** ÔÇö `persistTurn`'s post-turn `this.index.appendDailyLog(...)` (snippet of user message) was a third caller that the original plan didn't enumerate; routed through `appendToDailyLogSafely` so the grep acceptance check holds.
- **test/memory-write-safely.test.ts** ÔÇö new. 6 vitest cases covering: clean write, instruction-injection block, marker-injection block, threshold-raise-still-blocks, audit-mode-no-throw-and-logs, and `MemoryWriteBlocked` field shape.

## Surprises / scope concerns
- **Plan ┬ž1B enumerated 4 call sites; reality had 5.** `src/memory/manager.ts:185` had a fifth direct `appendDailyLog` call (post-turn user-message snippet) that the audit's F5 description and the plan both missed. Routed it through `appendToDailyLogSafely` because the grep acceptance check is unambiguous ("zero hits outside `write-safely.ts`"). Treat as a corollary of F5 rather than scope creep.
- **`checkMemoryTaint` is gated on score, not on the `safe` flag.** The current `sanitize.ts` hardcodes its 0.3 cutoff inside `checkMemoryTaint`. The task spec implies threshold is a per-call argument, so `applyGateChain` blocks on `taint.injectionScore >= threshold` and ignores `taint.safe`. Behavior parity with the old inline gate at default `threshold=0.3`; callers that raise it (none today) get the looser behavior they asked for. External-content markers still block at any reasonable threshold because they set `injectionScore = 0.95` in `sanitize.ts`. Did not touch `sanitize.ts` per the constraint.
- **`runMemoryGate` is a fourth export beyond the spec's single `writeMemorySafely`.** Reason: `memory_save target="retain"` writes to the DB via `memory.retain`, not to a file. The chain has to run before that DB write or retain becomes the new bypass. `runMemoryGate` does the chain and returns sanitized content for callers that own their own sink. Kept the API surface minimal ÔÇö one extra named export, no flags or modes.
- **Backup `.bak` writes (full_replace) now run through the gate too.** Existing content was already gated on its way in, so this is double-gating. The patterns are idempotent on already-clean text, so the only failure mode is "existing IDENTITY.md somehow already poisoned" ÔÇö in which case we correctly refuse to back it up rather than create a corrupted .bak file. Wrapped in `try {} catch {}` to preserve the original "backup failures are silently swallowed" behavior.
- **File size**: `write-safely.ts` is 193 LOC (under the 200 LOC ceiling). `save.ts` grew from 244 Ôćĺ 265 LOC. `auto-extract.ts` grew from 87 Ôćĺ 137 LOC (six try/catch blocks for the per-target error paths). Both still well under the 400 LOC project limit.

## Adjacent findings noticed but NOT fixed (per scope rules)
- **`checkMemoryTaint` exposes `safe: boolean` but its internal cutoff is hardcoded.** It can't be raised, only lowered (by the consumer). Should probably take a `threshold` parameter so the API matches its semantics; that's a `sanitize.ts` change and the task explicitly forbade touching it. Filed mentally as a follow-up to F5 ÔÇö not blocking.
- **`MEMORY_INJECTION_PATTERNS` is a strict subset of `INJECTION_PATTERNS`** (12 vs 40+). The audit (Drift evidence on F5) called this out. Out of scope for this task ÔÇö funneling, not pattern unification. The funnel makes the unification trivial when it lands: it's a one-file change in `sanitize.ts`, with no caller fan-out.
- **`MemoryIndex.appendDailyLog` and `MemoryIndex.writeMemoryFile` are now only called from `write-safely.ts`.** Could move the methods into `write-safely.ts` and drop them from `index-core.ts`, but that would mean exposing `markDirty` and the private `reindexThroughUniversal` hook. Decided keeping them as the underlying writer (called only via the funnel) was less invasive than relocating index internals. Flag for a future cleanup pass if the team prefers stricter encapsulation.
- **Parallel-task collisions:** sibling sessions also modified `src/canonical-loop/*` (task 2A ÔÇö terminal states), `src/security/credentials.ts` and `src/hooks/hook-engine.ts` (task 2D ÔÇö credential patterns), `src/providers/*` (1A), and `src/voice/voice-session.ts`/`gpu-session.ts` (3B.1). Did not touch any of those; staged edits visible in `git status` are not from this session.
