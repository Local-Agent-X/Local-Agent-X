## Files changed
- `src/tools/pre-dispatch.ts` — new (~140 LOC). Exports `assertToolCallAllowed(call, ctx)` + `ToolBlocked` error class with `stage`/`reason`/`recovery`. Each gate (session-policy, security, RBAC, tool-policy, threat preCheck, approval) is conditional on its ctx field, so the same function serves both dispatch paths without forcing either to ship gates it doesn't use.
- `src/tool-executor.ts` — replaced the inline `secDecision/rbacBlocked/policyBlocked` chain (the old L405–444 block) with one `await assertToolCallAllowed(...)` wrapped in try/catch. `ToolBlocked` → `{ content, isError, metadata: { layer, recovery } }` matches the original blocked-result shape exactly, so downstream threat-engine evaluation + budgeting + PostToolUse hooks see identical input. Session-policy short-circuit at L362 and the approval gate at L571 stay inline (different return semantics + dependency on circuit-breaker/rate-limit running first — see §Surprises).
- `packages/arikernel/tool-executors/src/base.ts` — added `setPreDispatchGate(fn)` / `runPreDispatchGate(toolCall)`. AriKernel stays SAX-agnostic; the host injects the gate at boot.
- `packages/arikernel/tool-executors/src/{file,http,shell,database,retrieval,sqlite-database}.ts` — every concrete executor now calls `await runPreDispatchGate(toolCall)` as the first statement of `execute()`. Six edits, one line each.
- `packages/arikernel/tool-executors/src/index.ts` — re-exports `setPreDispatchGate` + `runPreDispatchGate`.
- `src/server/bootstrap-ari-gate.ts` — new. `wireAriPreDispatch(security, toolPolicy)` calls `setPreDispatchGate` with an adapter that maps AriKernel `{toolClass, action}` → SAX tool-name and invokes `assertToolCallAllowed`. Discriminator goes the locked-down direction when ambiguous (http→`http_request`, not `web_fetch`).
- `src/server/bootstrap-services.ts` — calls `wireAriPreDispatch` after `security` and `toolPolicy` are constructed, before any tool dispatch can happen.
- `test/tools/pre-dispatch.test.ts` — new. 4 tests, covers: gate fires before executor body; gate throw blocks executor body (file content unchanged); `wireAriPreDispatch` routes the executor through security.evaluate (reads inside workspace pass, reads outside throw `BLOCKED by security`); `assertToolCallAllowed` throws `ToolBlocked` when approval is denied (`vi.spyOn` on `toolNeedsApproval` to bypass the user's `toolApproval: auto` setting).

## Surprises / scope concerns
- **The "inline chain" turned out to be three separate gate blocks, not one.** L362 (session-policy, returns immediately), L405–444 (security/rbac/toolPolicy, falls through to post-checks), L571–586 (approval, returns immediately, after circuit-breaker / rate-limit / hooks). Lifting only L405–444 preserves "no behavior change for chat-path (same checks, same order)" strictly. Lifting all three would have either reordered approval ahead of circuit-breaker/rate-limit (semantic change) or required threading two callsites. The shared function still supports all five gates for callers that want them (AriKernel does); chat-path opts out via `skipSessionPolicy: true` + no `approval` ctx.
- **No threat pre-check exists in the chat-path today.** `evaluateToolResult` runs post-execution; the `isRestricted()` external-tool block also runs post-execution (overwriting `result`, not preventing the call). `assertToolCallAllowed` includes a `threatEngine.preCheck` slot (gated on `ctx.threatEngine`), but neither dispatcher currently passes one — keeps the door open for 2C without changing semantics now.
- **RBAC is not in the plan's named chain** (`security → sessionPolicy → toolPolicy → threat → approval`). I included it anyway as an optional ctx field so the chat-path can lift its RBAC check into the same function and preserve the original `security → rbac → toolPolicy` precedence. AriKernel callers don't pass RBAC — no effect on that path.
- **AriKernel package boundary** — `@arikernel/tool-executors` can't depend on `src/`. Resolved via a setter (`setPreDispatchGate`) in `base.ts` that SAX wires at boot. Standalone package use stays no-op (no gate set → `runPreDispatchGate` is a no-op). This is the "adapter at the call site over rewriting the layer" pattern the prompt called out as preferred.
- **AriKernel ToolCall → SAX tool-name mapping is ambiguous** for some shapes (http+get could be `http_request`/`web_fetch`/`browser`). The adapter in `bootstrap-ari-gate.ts` always picks the most-locked-down equivalent so policy errs strict. The chat-path's per-tool semantics flow through the chat-path's existing logic; this mapping only matters for callers that route directly through an AriKernel executor.
- **AriKernel package rebuild required** — first attempt left stale `dist/` because `scripts/build-ari.js` skips packages where `dist/index.js` exists. Cleared `packages/arikernel/tool-executors/dist` + `node_modules/@arikernel/tool-executors/dist`, then `npm run build:ari` picked up the new exports. Worth noting if other tasks edit AriKernel src — they need the rm dance.
- **Test config** — repo's `vitest.config.ts` only includes `test/**/*.test.ts`. The existing `src/*.test.ts` files (security.test.ts, tool-policy-argmatch.test.ts, etc.) aren't currently picked up — those files have pre-existing failures unrelated to 1C. Moved my test to `test/tools/pre-dispatch.test.ts` to match the convention without expanding test discovery scope.
- **`toolApproval: "auto"` in the user's env** disables approval globally, which surfaced when the first cut of test 3 (using the real `toolNeedsApproval`) silently passed approval through. Final test mocks `toolNeedsApproval` via `vi.spyOn` so the approval branch is exercised regardless of host config.

## Adjacent findings noticed but NOT fixed (per scope rules)
- **F2 / F4 (registry + dispatcher collapse)** — confirmed still split. The pre-dispatch chain is now shared, but the registries (`tools/registry-build.ts` vs `tool-search.ts` vs `arikernel/tool-executors/registry.ts`) and dispatchers (`tool-executor.ts:executeSingleTool` vs `Pipeline.intercept`) remain parallel. That's 2C.
- **`packages/arikernel/runtime/src/pipeline.ts:457`** — the Pipeline calls `executor.execute(toolCall)` AFTER its own policy-engine pass, so the new pre-dispatch fires twice in the SAX-via-ariEvaluate flow (once inside Pipeline's policy-engine, once via the gate at the executor body). Currently harmless because chat-path registers no-op executors with Ari (per `src/ari-kernel.ts:53-72` — Ari is observer + gate, real exec runs through the chat-path's own tool dispatcher). If a future SAX path uses Ari's real executors directly, the double-check is still strictly safer, just slightly redundant. 2C's dispatcher collapse subsumes this.
- **Chat-path approval still inline at L571.** A future task could pull it into a single `assertToolCallAllowed` call by accepting a small reorder (approval moves ahead of circuit-breaker/rate-limit/hooks). Strictly-safer behavior change — user gets the prompt earlier, gets BLOCKED if rate-limited after approving. Left for 2C since it's a behavior change, not a refactor.
- **`evaluateToolResult` + post-execution `isRestricted` block at L626–660** — could move pre-execution into `assertToolCallAllowed`'s threat slot once chat-path is comfortable with the order change. Same reasoning as above — deferred to 2C.
