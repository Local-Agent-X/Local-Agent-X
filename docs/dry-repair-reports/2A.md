## Backfill / migration

Persisted record on disk: `~/.lax/agent-runs/<id>.json` written by `AgentRunStore.save`. Pre-F13 records carry `status: "done" | "error" | "timeout"`.

**Strategy chosen: read-side normalisation, no on-disk rewrite.**

- `AgentRunStore.get` and `AgentRunStore.list` now run every read through `migrateRunStatus`: `done → succeeded`, `error → failed`, `timeout → failed + reason: "timeout"`. Idempotent.
- `AgentRunStore.list({ status })` runs the query string through `migrateStatusFilter` so an older UI that asks `?status=done` still finds migrated records.
- Existing JSON files are not touched. When `save` rewrites a record post-completion it lands with the new vocabulary; the read normaliser quietly cleans up everything else on access.

Rationale: agent-run records are append-only-ish (rarely re-saved after terminal). A one-shot disk migration would have touched every file in `agent-runs/` and offered no win beyond what `migrateRunStatus` already does on read. Read-side normalisation keeps the migration reversible (just revert the helper) and avoids irreversibly mutating user data on first boot post-deploy.

Pre-F13 stream-event types like `{ type: "done" }` from adapter / SSE streams are NOT run-state and were left alone — they are a separate vocabulary (chunk-stream protocol) and the audit's grep acceptance only targets run-state literals.

## Surprises / scope concerns

- **`AgentStatus` in `src/agency/types.ts` was load-bearing for Handler-side status writes** — the Handler's FieldAgent uses it directly, and the `Run` doc comment promised `FieldAgent` is "structurally compatible with `Run`". To keep that comment honest after the `Run` swap, I had to swap `AgentStatus` too. Handler has no cancel/timeout statuses (its cancel path conflates with `error → failed`), so the Handler-side set is `{idle, working, waiting, succeeded, failed}`. The behavior smell of "cancel writes failed" is real but explicitly out of scope — F1 / task 2B retires the Handler entirely; F13 just stops the words drifting.

- **Three Handler-internal callbacks (`AgentUpdateCallback.type = "output" | "status" | "complete" | "error"`) were NOT touched.** That `"error"` literal is the update-event-type discriminator, not a run-state value. Same for `EventBus.emit("handler:agent-error", …)` event names. Cleanly distinct from run vocabulary.

- **CSS class names on the IDE agent card (`'done'`, `'error'`, `'working'`) were kept.** Those are stylesheet keys; the run-state comparison that drives the boolean (line 39–40) was migrated. Bare-status display in the chat-agent-feeds card now prints "succeeded" / "failed" literally — matches the canonical-loop bus vocabulary, which is also what users will see in the worker-card status pill going forward. Added `.agent-feed-card.succeeded` as a peer of `.done` in `app.css` so the styling doesn't regress.

- **Read-side migration instead of one-shot file rewrite** — explained above. If reviewers prefer an on-disk migration, the lift is small (iterate `agent-runs/*.json`, apply `migrateRunStatus`, write back). I'd run that as a separate commit; not in this one.
