# 2B: Route `invokeAgent` + primal-auto-build through `runAgentViaCanonical`
Closes: F1
Status: completed

## What this commit removes

The shadow agent runtime in `src/agency/handler.ts`. Before this commit:

```
invokeAgent → invokeDefinition → Handler.spawnAgent → Handler.runAgentAsync
   → EventBus.emit("handler:agent-run") → handler-events.ts subscriber
   → runAgentViaCanonical → terminal → EventBus.emit("handler:agent-result")
```

Two state machines tracked the same run — canonical's op record (persisted,
recoverable from `~/.lax/operations/<opId>/events.jsonl`) and Handler's
in-memory FieldAgent (lost on crash). The Handler's state machine added no
information canonical didn't already carry; it just held the cancel
AbortController and emitted the EventBus signals legacy subscribers consume.

After this commit, `invokeDefinition` calls the canonical-loop driver
directly and the Handler is a thin FieldAgent registry that holds the
AbortController + supports the legacy status / cancel / message tools —
nothing more.

## Files changed

- `src/agents/runtime.ts` — **new**. Registers a single `AgentRunDriver`
  during server startup; `dispatchAgentRun(req, signal)` looks up the
  registered driver and awaits its result. The driver is a function with
  signature `(AgentRunDriverRequest, AbortSignal) → Promise<{result, success,
  tokens?}>`. Decouples `src/agents/invoke.ts` (pure module, no server
  state) from `src/server/handler-events.ts` (owns config / secrets / tool
  policy / project rosters / runAgentViaCanonical import) without forcing
  an `agents/ → server/` import cycle.
- `src/agents/invoke.ts` — rewritten so `invokeDefinition`:
  1. Caps tools through project gate + override (unchanged).
  2. Calls `Handler.attachExternalRun(...)` to mint the agentId + register
     a FieldAgent record for `agent_status` / `agent_cancel` / `agent_message`
     to find later.
  3. Emits `handler:agent-spawn` synchronously so the UI broadcaster +
     `pendingMeta` capture in `handler-events.ts` fire (drives the History
     tab live-merge and the AgentRunStore record build-up).
  4. Fires the registered driver async via `runAgentViaDriver(...)` —
     translates the driver's terminal outcome into `handler:agent-done` /
     `handler:agent-error` (chunk-runner subscribes here) and the durable
     `handler:agent-result` (the AgentRunStore.save trigger).
  5. Returns the same `RunRef` shape as before. `runId` and `fieldAgentId`
     are both the synthesised "field-agent-N-..." id so legacy code that
     pattern-matches the id shape is unaffected.
- `src/agency/handler.ts` — added two methods + two `@deprecated` markers:
  - `attachExternalRun(config)` — mints "field-agent-N-..." id, adds
    FieldAgent in `working` state, subscribes to the message bus, returns
    `{agentId, abortController}`. Does NOT emit `handler:agent-spawn` (the
    caller does, so ordering stays spawn → run → result) and does NOT run
    anything (the caller owns the canonical-loop dispatch).
  - `finalizeExternalRun(agentId, outcome)` — sets terminal status, appends
    output, fires `notifyUpdate`, pushes completion to the parent session,
    and schedules the 5-min cleanup from the FieldAgent map. Mirrors the
    legacy `runAgentAsync` post-run lifecycle exactly so the only thing
    that changes from a consumer's POV is who drives the run.
  - `@deprecated` on `Handler.spawnAgent` and `Handler.runAgentAsync`.
    Both stay in place — the comment names the replacement
    (`runAgentViaCanonical` / `invokeDefinition`) and the deletion target
    (one release after 2026-05-13). No production caller remains; the
    methods exist only for legacy test imports.
- `src/server/handler-events.ts` — the `eventBus.on("handler:agent-run")`
  subscriber is replaced by `registerAgentRunDriver(agentRunDriver)`. The
  driver body is the same prep (parent context, briefing, identity block,
  worktree provisioning, tool resolution by audience) + `runAgentViaCanonical`
  call that used to live in the subscriber, with one structural change:
  it RETURNS the terminal `{result, success}` instead of emitting
  `handler:agent-result` itself. `invokeDefinition`'s `runAgentViaDriver`
  bridges the return value to the EventBus emission. The other four
  subscribers (`handler:agent-spawn`, `handler:agent-output`,
  `handler:agent-blocked`, `handler:agent-result`, `handler:agent-redirect`)
  are unchanged. One small correctness improvement rode along: the driver
  now respects an invocation's resolved tool list when no template is
  attached, instead of silently dropping it — closes a tools-override drop
  that affected operations/executor's phase agents.
- `src/operations/executor.ts` — `spawnPhaseAgent` no longer calls
  `Handler.getInstance().spawnAgent(...)`. Builds an inline AgentDefinition
  (id `inline-operator-<opId>-<phaseId>`) and calls `invokeDefinition`. The
  EventBus result-wait pattern around it is unchanged — the agentId still
  flows back as `ref.fieldAgentId`, and `handler:agent-result` still fires
  on terminal.
- `src/routes/agents.ts` — `POST /api/agents/templates/:id/spawn` builds an
  AgentDefinition from the AgentTemplate and routes via `invokeDefinition`.
- `src/primal-auto-build/agents/chunk-runner.ts` — comment refresh only.
  No code change (the module already called `invokeDefinition`; only the
  doc header referred to the deprecated "Handler → adapter" path).
- `test/agents-runtime-canonical.test.ts` — **new**. 7 tests covering:
  - Driver dispatch: receives the resolved `AgentRunDriverRequest` and an
    AbortSignal that isn't pre-aborted.
  - Event-bridge: `handler:agent-result` fires with `{agentId, result,
    success, tokens?}`; `handler:agent-done` fires on success;
    `handler:agent-error` fires on `success: false`.
  - Spawn parity: `handler:agent-spawn` fires synchronously with the
    minted agentId so the UI broadcaster + History live-merge keep
    working.
  - Handler registry: an in-flight invoke is visible via
    `Handler.getAgentStatus(agentId)` — closes parity for `agent_status`.
  - Cancel parity: `Handler.cancelAgent(agentId)` fires the AbortSignal
    the driver received — closes parity for `agent_cancel`.
  - Error parity: a driver rejection becomes a `success: false` result,
    not a hang.
  - chunk-runner subscriber: `handler:agent-done` carries the result string
    the chunk-runner.ts subscriber depends on.

## Persistence parity

The audit's bite: "an agent run that crashes inside primal-auto-build has
no recovery path." After 2B:

1. Every `invokeDefinition` call hits the registered canonical-loop driver
   in `server/handler-events.ts`.
2. The driver calls `runAgentViaCanonical` which, inside, calls `writeOp`
   (op record → `~/.lax/operations/<opId>/operation.json`) and routes the
   loop through `canonicalLoopEntry` (which writes `events.jsonl` events
   transactionally via `appendEvent` / `appendOpEvent`).
3. State transitions, tool calls, turn boundaries, and terminal state all
   land in `events.jsonl` before the next iteration is scheduled.

So if LAX crashes mid-tool-call during a primal-auto-build chunk worker:
- `~/.lax/operations/<opId>/operation.json` carries the op state.
- `events.jsonl` carries every state transition + tool-call event.
- `opEventsSince(opId, -1)` replays the full history; `reconnectOp` attaches
  to the live bus with seq-deduped replay.

This is exactly what `canonical-loop-04-reconnect-replay.test.ts` already
pins (PRD test #9), and `canonical-loop-08-lease-and-crash-recovery.test.ts`
covers the lease-expiry / supervisor-recovery side. Every agent invocation
post-2B rides on that contract — that's the parity F1 demanded.

The 2B-specific test (`test/agents-runtime-canonical.test.ts`) demonstrates
the *dispatch* parity (invokeDefinition → registered driver → terminal
events). The persistence + replay primitives have their own dedicated tests
and don't need to be reproduced at the invoke layer.

## Surprises / scope concerns

- **`runAgentViaCanonical` already existed on the sub-agent path.** The
  `runagent-caller-inventory.md` from P4.C1 documents this: the
  `on("handler:agent-run")` subscriber was already calling
  `runAgentViaCanonical` since the P4.C4 sub-agent migration. The shadow
  state machine F1 flagged was Handler's own FieldAgent registry +
  EventBus round-trip — NOT a separate model loop. 2B's diff is therefore
  smaller than "build a new canonical driver"; it's "stop pretending the
  Handler hop adds anything and let invoke.ts drive canonical directly."
  The Handler stays as a FieldAgent registry for the legacy status/cancel
  tools, and `runAgentAsync` is deprecated, not deleted, per the plan.

- **Tools-override drop in the legacy driver — fixed in passing.** Pre-2B
  `on("handler:agent-run")` only respected `template.allowedTools` when
  resolving spawnable tools; the invocation's `tools` field from
  `SpawnConfig.tools` was dropped silently. Operations/executor's phase
  agents passed `phaseObj.suggestedTools` and got the default
  spawned-agent surface instead. The new driver respects
  `req.tools` when no template is supplied. Behavior change is restricted
  to ad-hoc invocations (no template); template-driven runs still resolve
  from `template.allowedTools` first. Noted because a careful reviewer
  will spot the if-branch.

- **`Handler.cancelAgent` writes `failed`, not `cancelled`.** Pre-2B
  behavior. 2B preserves it — the FieldAgent's terminal vocabulary is
  `{succeeded, failed}` per `AgentStatus` (2A's terminal-state alignment
  noted this), and the canonical op transitions through `cancelled` but
  the bridge maps that to a `success: false` for the legacy subscribers.
  Closing this fully needs `AgentStatus` to gain `cancelled` and the
  `cancelAgent` finaliser to write it — out of 2B scope (touches the
  `AgentRun.status` migration 2A already shipped read-side normalisation
  for). Filed below.

- **`Handler.spawnAgent` still works.** Marked `@deprecated`, not deleted.
  No production caller remains in `src/` (grep is clean). Per the plan,
  deletion is a follow-up task one release after 2026-05-13.

- **`handler.ts` is now 552 LOC.** The file was 451 LOC pre-2B (already
  over the 400-LOC limit per CLAUDE.md / AGENTS.md). 2B adds ~100 LOC
  (`attachExternalRun` + `finalizeExternalRun` + deprecation comments).
  Splitting handler.ts into separate registry / lifecycle / message
  modules is the right architectural move, but it's explicitly outside
  2B scope and pairs naturally with the upcoming `runAgentAsync` deletion
  (which removes ~150 LOC and brings the file back near the limit).

## Adjacent findings noticed but NOT fixed

- **Pre-existing threat-trust-ledger / threat-engine-user-consent test
  failures.** Same 4 failures on `main` HEAD as on `dry-repair/2B`
  (verified by stash + checkout). Test-state leakage between tests in
  the same file — `recordApproval` mutations bleed across the
  `beforeEach`. Pure test-isolation bug, no production code reason.
  File a separate ticket; not in 2B's scope.

- **chunk-runner's `signal` doesn't propagate into the canonical op.**
  `runChunkAgent`'s `opts.signal` shortcuts its own Promise on abort
  but doesn't pass through to `invokeDefinition` (which would need an
  `InvokeOpts.signal` to forward to `Handler.attachExternalRun`'s
  AbortController). So a parent-loop cancel from primal-auto-build
  short-circuits the local wait but leaves the canonical op running
  in the background until its own wall-clock-ceiling expires. Pre-2B
  behavior — flagged because the F1 mandate is recoverable + cancellable
  runs, and this is a partial cancel. The fix is a 3-line addition to
  `InvokeOpts` + `Handler.attachExternalRun`; out of 2B because it adds
  scope ("propagate cancel end-to-end") beyond "route the dispatch."

- **`Handler.cancelAgent` → `failed` vs `cancelled`.** See above.
  Cleanly resolved by the F1 follow-up (delete `runAgentAsync` + drop
  the FieldAgent state machine entirely, since canonical owns the
  `cancelled` terminal already).

- **The 5-min FieldAgent map TTL still exists in `finalizeExternalRun`.**
  Same value the legacy `runAgentAsync` used. Matches existing UI assumptions
  (History tab live-merge expects in-flight entries to age out). Don't change.
