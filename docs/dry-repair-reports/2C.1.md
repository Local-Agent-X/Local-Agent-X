# 2C.1 — Unify the tool registry

**Closes:** DRY-AUDIT.md F2 (sub-commit 1 of 3 for Phase 2C). Scope is registry only — dispatcher unification ships in 2C.3.

**Branch:** `dry-repair/2C.1` (off `main` @ `128e6ca`).

## What changed

Three drifting registries collapsed into one indexed store.

### New: [src/tools/registry.ts](../../src/tools/registry.ts)

Single source of truth, defined as `UnifiedToolRegistry` plus a process-wide `unifiedRegistry` singleton. Each entry now carries:

- Chat-path metadata: `tool` (ToolDefinition), `defer`, `tags`, `searchHint`
- AriKernel metadata: optional `toolClass` (the existing `@arikernel/core` `ToolClass` enum) so the AriKernel side can filter
- MCP metadata: optional `mcpSource` (server name) so MCP-sourced tools are addressable as a group

Public API (intentionally a superset of the old `ToolRegistry` so call sites don't break):

| Method | Purpose |
|---|---|
| `register(tool, opts)` | Insert / overwrite (last write wins — matches MCP reload semantics) |
| `unregister(name)` | Drop an entry by name |
| `get(name)` / `getEntry(name)` | Name-keyed lookup (chat-path view) |
| `getAll()` | All registered tools |
| `getEagerTools()` / `getDeferredTools()` | Eager vs deferred split (used by request resolver + `tool_search`) |
| `getByToolClass(toolClass)` | **AriKernel view** — tools tagged with a given `toolClass` |
| `getMcpTools(serverName?)` | All MCP-sourced tools, optionally filtered by server |
| `search(query, max)` | Keyword search backing the `tool_search` tool |

File size: 188 LOC (well under the 400 cap).

### Delegated: [src/tool-search.ts](../../src/tool-search.ts)

The local `ToolRegistry` class and `RegistryEntry` interface are gone. Replaced with a re-export from the unified module:

```ts
export { UnifiedToolRegistry as ToolRegistry } from "./tools/registry.js";
export { unifiedRegistry } from "./tools/registry.js";
```

Existing call sites that `import { ToolRegistry } from "./tool-search.js"` continue to compile and run unchanged. `resolveToolsForRequest`, `IDENTITY_TOOLS`, `toolSearchEnhancements`, and `createToolSearchTool` remain in this file — they're audience-resolution logic, not registry logic.

### Delegated: [src/tools/registry-build.ts](../../src/tools/registry-build.ts)

`new ToolRegistry()` is gone. The module now uses the `unifiedRegistry` singleton. `buildToolRegistry()` keeps the same return shape (`{ registry, eagerTools, toolSearchTool, promptSection }`) so `bootstrap-tools` and downstream consumers don't care.

### Demoted: [packages/arikernel/tool-executors/src/registry.ts](../../packages/arikernel/tool-executors/src/registry.ts)

`ExecutorRegistry` no longer self-constructs the five default executors in its constructor. It's now a passive `Map<toolClass, ToolExecutor>` filled from the outside. The historical loadout is recoverable via:

```ts
ExecutorRegistry.withDefaults()   // http + file + shell + database + retrieval
```

The `Firewall` constructor in `packages/arikernel/runtime/src/firewall.ts` switched to `ExecutorRegistry.withDefaults()` so its public behavior is unchanged. Tests that depend on the five defaults being present continue to pass.

The "view" semantics from the task spec land here: chat-path tools live in `unifiedRegistry`, and `getByToolClass()` is the SAX-side view of "which chat-path tools are tagged with this AriKernel class." The dispatcher unification (2C.3) is what bridges chat-path `ToolDefinition.execute(args)` to AriKernel `ToolExecutor.execute(toolCall)` — that work is intentionally out of scope here.

### MCP catch-up workaround removed: [src/server/bootstrap-tools.ts](../../src/server/bootstrap-tools.ts)

Before:

```ts
const mcpTools = mcpManager.getAllTools();
allAgentTools.push(...nonRedundantMcpTools);
// ... later in the same function ...
for (const tool of allAgentTools) {
  if (!toolRegistry.get(tool.name)) {
    toolRegistry.register(tool, { defer: true, ... });
  }
}
```

After:

```ts
for (const tool of nonRedundantMcpTools) {
  const serverName = tool.name.replace(/^mcp_/, "").split("_")[0] ?? "unknown";
  toolRegistry.register(tool, {
    defer: true,
    tags: ["mcp", serverName],
    searchHint: tool.description.slice(0, 80),
    mcpSource: serverName,
  });
}
```

MCP tools land in the unified registry at the moment they're fetched, with their originating server name preserved as both a tag and a structured `mcpSource` field. The post-pass loop still indexes the runtime-built non-MCP tools (memory, secret, browser, image, protocol, cron, agent, agency, app, issue, operation), but there's no longer a separate MCP-specific catch-up step.

## Reversibility

Every callsite still imports `ToolRegistry` from `tool-search.js`. Reverting this commit restores the parallel-registry world without ripple changes elsewhere. No public API was renamed.

## What did NOT change (left for 2C.2 / 2C.3)

- The four parallel policy layers (F4) — that's 2C.2.
- The two parallel dispatchers (F2 part 2 / chat-path `executeSingleTool` vs AriKernel `Pipeline.intercept`) — that's 2C.3.
- AriKernel's `ExecutorRegistry` is still a separate physical store from the unified registry. The unified registry holds chat-path `ToolDefinition` shapes; the AriKernel side holds `ToolExecutor` instances. Bridging the two execution shapes is a 2C.3 concern. Today, the unified registry is the source of truth for *metadata + which tools belong to which `toolClass`*; the AriKernel `ExecutorRegistry` continues to hold the actual executor instances via `withDefaults()`.
