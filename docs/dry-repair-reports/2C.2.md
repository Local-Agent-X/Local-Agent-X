## Surprises / scope concerns
- The four "policy layers" in the original audit have very different surface shapes:
  - `SecurityLayer.evaluate` takes a `ToolCallContext` and runs file/shell/network helper functions.
  - `ToolPolicy.evaluate` runs declarative `ToolPolicyRule[]` (with argMatch, constraints, priority).
  - The "threat-engine pre-check" wasn't actually a `threatEngine.preCheck()` method — it was an inline `RESTRICTED_EXTERNAL_TOOLS` block inside `assertToolCallAllowed`. I extracted it into `makeThreatEnginePack`.
  - `packages/arikernel/policy-engine` evaluates `(ToolCall, TaintLabel[], Capability[])` — totally different signature from SAX tool-call dispatch.

  Pragmatic resolution: each pack carries (a) a **descriptive `rules: RulePackRule[]`** (Zod-shaped, for introspection / future admin UI) and (b) a **pack-internal `evaluate(call, ctx)`** that wraps the existing implementation. Rule SEMANTICS are unchanged. The Zod schema describes the rule surface; the evaluator dispatches uniformly.
- The `arikernel` pack is effectively a no-op at runtime (returns allow) because SAX's `DEFAULT_POLICY.defaultDecision = "deny"` already enforces the same deny-by-default posture that AriKernel's `DENY_ALL_RULE` provides. I considered making the arikernel pack actually wrap `new PolicyEngine().evaluate(...)` but the shape mismatch (it needs ToolCall with taintLabels + Capability[]) would have required either a stub adapter that always passes a "shell.exec" capability, or rewriting that signature — both are scope creep ("don't change rule semantics"). The descriptive `rules` field carries DENY_ALL_RULE so audit / introspection still sees it; the runtime `evaluate` is a no-op allow, with the deny-by-default actually enforced by the default-policy pack (priority 20, which runs before arikernel at priority 90 in the priority order, so arikernel never decides anything in practice). This matches the constraint: "Don't change rule semantics. The evaluator is a dispatcher, not a rule rewriter."
- Ordering note: in the old pre-dispatch chain, RBAC ran AFTER security and BEFORE tool-policy. In the new code, RBAC runs BEFORE the evaluator (i.e., before security). Both are denies — order only matters for which deny REASON is reported when both would deny. RBAC-first is arguably more correct (it's a principal property; cheaper to check first), and no existing test asserts on the order. Flagging in case downstream code did depend on the old order.
- **No rule conflicts found between src/tool-policy/default-rules.ts and packages/arikernel/policy-engine/src/defaults.ts.** SAX default-policy is `{defaultDecision: "deny", rules: [allow-listed tools…]}`. AriKernel defaults are `[DENY_ALL_RULE]`. Both are deny-by-default. No "stricter one" decision needed; they agree on posture.

## Adjacent findings noticed but NOT fixed
- **packages/arikernel/runtime/src/pipeline.ts:353** still calls `this.policyEngine.evaluate(toolCall, inputTaints, this.principal.capabilities)` directly. That's AriKernel's INTERNAL pipeline doing its own policy enforcement on its own typed inputs (Capability/TaintLabel). It's not a SAX dispatch path — SAX calls AriKernel-side tools through `bootstrap-ari-gate.ts → assertToolCallAllowed → evaluator`. The two paths now share SAX's unified evaluator on the SAX side; AriKernel's internal pipeline still runs its own engine. Unifying THAT layer would mean either (a) collapsing AriKernel's pipeline into SAX or (b) teaching the evaluator about Capability/TaintLabel — both are 2C.3 territory ("collapse the dispatcher") or beyond.
- **src/hooks/hook-engine.ts:121** calls `security.evaluate({ toolName: "bash", … })` directly when running hook commands. Hooks aren't tool calls in the dispatcher sense, but they DO execute shell with the agent's authority. If the goal is one canonical policy decision per shell execution, hooks should also route through the evaluator. Out of scope for F4 (which is the four named layers), but a clean follow-up.
- **The `decision: "confirm"` verdict** in DEFAULT_POLICY (e.g., `flag-browser-evaluate`) maps to "allow + UI flag" in `ToolPolicy.evaluate`. The unified `RulePackRule` schema collapses it to "allow" since the evaluator's surface is binary allow/deny. The UI flag still works (it lives on the underlying `PolicyDecision.confirm` field returned by `ToolPolicy.evaluate`, which the default-policy pack reads). No behavior change, but the unified descriptor is lossy here. If admin UIs start reading `RulePackRule.decision` they'll miss the confirm flag — flagging for whoever builds those.
- **`makeArikernelPack` returns `{allowed: true}` unconditionally.** This is by design (see Surprises above), but it does mean the unification is somewhat cosmetic at the runtime layer — the arikernel pack exists as a descriptor in the rules list but doesn't actually gate anything that the default-policy pack didn't already gate. If F4 was intended to also enforce arikernel's capability/taint checks on SAX-side dispatch (rather than just unifying the SAX-visible deny-by-default posture), that's a real piece of work that needs Capability/TaintLabel context plumbed into `PolicyEvalCtx`. Treated as scope creep here; flagging.
