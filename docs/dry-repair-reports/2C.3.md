This is the highest-risk task in the plan. The change set is conservative-by-design — it preserves AriKernel's vendored safety stack rather than absorbing it, and surfaces the kernel-side executors as first-class SAX tools through one dispatcher instead of a parallel stack.

Input `ToolCall`, output `ToolResult` — the unified single-dispatcher contract from the plan. - **src/server/bootstrap-tools.ts:80-152** — registers the six bridge ToolDefinitions in `unifiedRegistry`, tagged with `toolClass` so `getByToolClass()` (the AriKernel-view from 2C.1) returns them. Adds them to `allAgentTools` so deferred discovery via `tool_search` works.

## pipeline.ts:353 outcome

**Routed.** The 2C.2 follow-up flagged this exact line as "AriKernel runtime still calls `policyEngine.evaluate` directly for its typed Capability/TaintLabel pipeline — 2C.3 territory." Resolution:

- Added module-level hook `setUnifiedPolicyPreCheck(fn)` to `packages/arikernel/tool-executors/src/base.ts`. SAX wires this in `bootstrap-ari-gate.ts:wireAriPreDispatch` with a callback that runs `evaluatePolicy(call, [securityPack, defaultPolicyPack, threatPack, arikernelPack], ctx)`.
- Added Step 3a in `pipeline.ts:354-371` that calls `runUnifiedPolicyPreCheck(toolCall)` before the existing Step 3 (now Step 3b at line 375). Deny propagates through the standard `ToolCallDeniedError` + audit log + `runState.recordDeniedAction()` path so accounting stays consistent.

Standalone consumers of the AriKernel package (vendored use case) leave the hook unset → pre-check is a no-op → previous behavior preserved exactly.

**The original line (`this.policyEngine.evaluate(toolCall, inputTaints, this.principal.capabilities)`) is still present at pipeline.ts:375** — it's the typed Capability/TaintLabel evaluator, and we deliberately keep it second. The unified evaluator (rule packs) doesn't take Capability[] / TaintLabel[]; the typed engine does. Series composition is the right move: unified packs gate "is this tool allowed at all under SAX's policy posture", typed engine enforces "does this principal hold a valid token for this capability class with these constraints". Both must pass.

## Surprises / scope concerns

- **`src/tool-executor.ts` and `packages/arikernel/runtime/src/pipeline.ts` are both pre-existing god files** (818 and 755 LOC respectively after this change; 767 and 733 before). CLAUDE.md / AGENTS.md cap is 400 LOC. Splitting them is outside 2C.3 scope ("don't add comments/types/cleanup to code you didn't touch unless needed for this task") and would expand the change set well past the soft cap. Flagging as separate follow-up.
- **Bridge adapter naming** (`ari_file` / `ari_shell` / etc.) is intentional — kept distinct from SAX's existing `read` / `bash` / `http_request` / etc. tools so the model doesn't get confused about which path it's calling. The bridge tools are deferred (only surfaced via `tool_search`), so the eager schema doesn't grow. SAX's main flow continues to use the existing tools; bridge tools are there for SAX-side callers who explicitly want the kernel's executor implementations (e.g., a test that wants `FileExecutor`'s O_NOFOLLOW path, a sub-agent that needs SSRF-safe HTTP through the kernel rather than SAX's `http_request`).
- **`packages/arikernel/policy-engine/src/engine.ts` PolicyEngine.evaluate signature** is unchanged. The unified-evaluator pre-check sits in front of it, not on top. If a future task wants to fully fold the typed engine into the unified evaluator, the prerequisite is wiring Capability[] / TaintLabel[] into `PolicyEvalCtx` — a different shape of change and one that should land separately so any regression has a clean revert.
- **AriKernel safety properties land on `ToolResult.metadata.arikernel`** as `{callId, durationMs, taintLabels, toolClass?}`. The plan said "FIELDS on the ToolResult envelope, not a separate execution stack." Putting them on `metadata` (the existing free-form bag) instead of as top-level fields keeps the SAX ToolResult shape stable — every existing tool already uses `metadata` for `duration_ms`, `recovery`, etc. Documented in the bridge's docstring.
- **`src/hooks/hook-engine.ts:121`** — explicitly not touched (the plan called this out as a separate concern). Hooks aren't tool calls; they're a different concern that needs its own task if/when consolidation happens.
