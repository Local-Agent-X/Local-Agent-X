# Task 2D Report — Credential pattern consolidation

**Date**: 2026-05-13
**Closes**: F11

## What changed

1. **New** `src/security/credential-patterns.ts` (60 lines) — single source for credential matching:
   - `CREDENTIAL_ENV_PREFIXES: RegExp` — env-var NAME prefixes (Anthropic / OpenAI / xAI / Cerebras / Groq / Mistral / Voyage / Google / Gemini / Azure / HF / GH / GitLab / Slack / Discord / Brave / Notion / Linear / Vercel / Stripe / Supabase / AWS / NPM / SMTP / IMAP / DeepSeek / Moonshot / Dashscope / VOICE_TOOLS / CUSTOM) plus generic `*_(KEY|TOKEN|SECRET|PASS|PASSWORD)$` suffix.
   - `CREDENTIAL_KEY_PATTERNS: readonly RegExp[]` — 22 inline secret-shape regexes (UNION of what was in `credentials.ts`).
   - `redact(text)` helper masks captured secrets via `value[:4] + "...[REDACTED]"`. Falls back to masking the entire match when a pattern has no capture group (closes a latent gap in the previous helper where PEM blocks passed through unchanged).
2. **Refactored** `src/hooks/hook-engine.ts` — `scrubEnv` now consumes `CREDENTIAL_ENV_PREFIXES`; the local `SCRUB_KEYS` regex is gone.
3. **Refactored** `src/security/credentials.ts` — now 13 lines (was 57). `redactCredentials(text)` delegates to `redact()` from the shared module. Module also re-exports `CREDENTIAL_KEY_PATTERNS` and `redact`.
4. **New** `test/credential-patterns.test.ts` (60 vitest cases):
   - 30 env-var names that should be flagged + 6 that shouldn't.
   - One fixture per inline pattern, asserting redact() rewrites the input AND leaves no plaintext tail.
   - `redactCredentials` and `redact` produce identical output on the fixture text.
   - Every pattern in `CREDENTIAL_KEY_PATTERNS` matches at least one fixture (guards against adding a pattern with no test).
   - Cross-scrubber check: a fake `sk-ant-...` value is caught by both `CREDENTIAL_ENV_PREFIXES.test("ANTHROPIC_API_KEY")` AND `redact()`.

## Behavior changes

- **PEM private-key blocks now actually get masked.** The previous `redactCredentials` had `/-----BEGIN ... PRIVATE KEY ... END-----/` in its pattern list but the callback returned `match` unchanged when no capture group existed, so PEM secrets passed through. The new helper masks the whole match in that case. Defensible improvement, not a regression.
- **Env-prefix list expanded.** New prefixes added vs. the prior `hook-engine.ts` list: `CEREBRAS_`, `GROQ_`, `MISTRAL_`, `VOYAGE_`, `GOOGLE_`, `AZURE_`, `HF_`, `HUGGINGFACE_`, `GH_`, `GITLAB_`, `NOTION_`, `LINEAR_`, `VERCEL_`, `STRIPE_`, `SUPABASE_`, `AWS_`, `NPM_`, `VOICE_TOOLS_`. Every prefix actually used as `process.env.*` in this repo is now covered.

## Constraints honored

- No changes to scrubber CALLERS. `scrubEnv` and `redactCredentials` keep their signatures and call sites untouched. The 4 known callers of `redactCredentials` (`memory/index-sync.ts`, `server-utils.ts`, `routes/settings/security.ts`, plus the `security.ts` re-export) compile and behave identically (other than the PEM masking fix above).
- Out of scope: `getCredential(name)` facade over vault/auth/env precedence (plan §2D defers).

## Discovered, out of scope

Two additional credential scrubbers exist with their own pattern lists; they are not in F11's stated scope and would each change semantics if collapsed:

- `src/workers/redactor.ts` — uses a `<redacted>` replacement format (op-log style), distinct from this module's prefix-masking format.
- `src/memory/utils.ts` — replaces with `[REDACTED]` whole-token, also includes credit-card and DB-URL patterns not in the security-side list.
- `src/secret-scanner.ts` — detection + classification (returns type/name labels for the `secret_scan` tool), not redaction.

Recommend a follow-up audit pass to decide whether `redactor.ts` and `memory/utils.ts` should consume `CREDENTIAL_KEY_PATTERNS` with their own replacement strategies, or stay separate by design.
