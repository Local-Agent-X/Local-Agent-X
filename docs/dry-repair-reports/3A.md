# Task 3A — Transport-Discriminated Provider Registry

**Closes**: F8 (Cerebras patch required 5 files — no provider registry) and F10 (Provider baseURL routing as hardcoded if-branch).

**Branch**: `dry-repair/3A` (off main).
**Commit**: `refactor(providers): transport-discriminated registry (closes F8, F10)`.

## What changed

### New module
- `src/providers/registry.ts` — single source of truth. Exports `PROVIDERS: Record<ProviderId, ProviderMeta>` where `ProviderMeta` is a discriminated union on `transport`:
  - `"http"` — `{ id, label, models, defaultModel, baseURL, envKey, capabilities }`. Used for the OpenAI-compat family (xai, openai, codex, gemini, cerebras, local, ollama-cloud, custom).
  - `"cli"` — `{ id: "anthropic", label, models, defaultModel, cliBinary, capabilities }`. No `baseURL`, no `envKey`. The Claude CLI subprocess owns auth.
- Adds helpers `isHttpProvider(meta)`, `resolveBaseURL(id, ctx)`, `isReasoningModel(id, model)`.
- `baseURL` is either a literal string or a context-resolving function (covers runtime-resolved providers: `local` uses `ollamaUrl`, `custom` uses `customBaseURL`). Functions can return `null` when required config is absent.
- Per-provider `capabilities.reasoning: RegExp | false` replaces the global `REASONING_CAPABLE` regex from `openai-http.ts:21`. Reasoning is now a property of the provider's models, not a single regex pretending to know about all of them.

### Call-site swaps
- `src/agent-request/resolve-provider.ts`:
  - `hasCredsFor(p)` discriminates on transport (anthropic via `loadAnthropicTokens()`, codex via `loadTokens()`, everything else via `secretsStore.get(meta.envKey)`).
  - API-key resolution routes by transport first (CLI/anthropic short-circuits), then by provider-specific quirks (codex OAuth, openai legacy fallback, custom `customBaseURL`), then a generic `secretsStore.get(meta.envKey)` for the rest.
  - Default model: `meta.defaultModel || config.model`.
- `src/routes/settings/providers.ts`:
  - Provider list / labels / model arrays derived from `PROVIDERS` via a `pushFromRegistry(id)` helper.
  - The `DEFAULT_MODEL` dict (10 lines, drift target) collapses to `PROVIDERS[id].defaultModel`.
  - Adds `GET /api/providers/registry` — static metadata endpoint for the UI.
- `src/canonical-loop/adapters/openai-compat.ts:533`:
  - The `if (provider === "cerebras")` / `xai` / `openai` / `gemini` / `local` / `custom` / `ollama-cloud` chain collapses to: transport-guard (anthropic → null), ollama-cloud carve-out (cache-pairs baseURL with apiKey, so it stays separate), then `meta.baseURL` resolution.
  - The transport discriminator IS the safety belt — anthropic cannot accidentally be routed through openai-compat.
- `src/providers/adapters/openai-http.ts:21`:
  - `REASONING_CAPABLE` regex removed. Replaced with `isReasoningCapable(baseURL, model)` that finds the matching registry entry by baseURL prefix and consults its `capabilities.reasoning`. Falls back to OSS-pattern matching for unknown baseURLs (local Ollama variants, custom endpoints).

### UI: chose Option A (under-30-LOC budget held)
- `public/js/apps.js`:
  - Hardcoded `APPS_PROVIDERS` and `APPS_MODELS` constants replaced with a single `fetch('/api/providers/registry')` on init. Two constants collapse into one network read.
  - **Diff**: −22 LOC, +12 LOC (net −10).
- **`public/app.html` LEFT ALONE** (per Option B partial-flag). The Apps gallery dropdown is now registry-driven; the legacy settings page dropdown at `app.html:287` and `settings-providers.js`'s `PROVIDER_KEY_CONFIG` still hardcode their own copies. Flagging this as known-stale shadow until the UI rebuild lands. Replacing those would have pushed the UI change over the 30-LOC threshold and tangled with `settings-providers.js`'s per-provider key-config UI that this task is not scoped to.

The acceptance check ("zero hits outside registry.ts for provider-name metadata switches") passes for the targeted metadata files. The remaining behavioral switches are a different DRY problem and not part of F8/F10.

## Done when

Adding a new provider `groq` requires editing **only** `registry.ts` for the backend:
- baseURL → routed via `resolveOpenAICompatTarget` (no edit).
- envKey → routed via `hasCredsFor` / api-key resolution (no edit).
- default model → routed via `pushFromRegistry` / `meta.defaultModel` (no edit).
- UI dropdown in Apps gallery → routed via `/api/providers/registry` fetch (no edit).
- Reasoning capability → routed via `capabilities.reasoning` regex (no edit).
- Type union → already derived from `PROVIDER_IDS` (1A) — one edit at `src/providers/provider-ids.ts`.

The UI in `app.html` line 287 and `settings-providers.js`'s `PROVIDER_KEY_CONFIG` still need touching for a new provider's settings UI. Flagged as follow-up.

## Reversibility

Trivial. The registry shape is additive; reverting any of the call sites just restores their inline metadata. The transport discriminator is the only structural choice — and the audit explicitly calls it out as the right shape, so reverting that would be the regression.
