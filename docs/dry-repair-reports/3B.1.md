## What changed

1. **`integrations/open-voice/bridge.ts:131`** — `isOpenVoiceBridgeEnabled()` now returns `process.env.LAX_VOICE_OPEN !== "0"`. Unset env → open-voice path. Explicit `LAX_VOICE_OPEN=0` → legacy path (the escape hatch).
2. **`integrations/open-voice/bridge.ts:11-15`** — header comment updated; the old comment claimed legacy was the default "for one release," which is no longer true.
3. **`src/voice/voice-session.ts:218-222`** — `createVoiceSessionFactory()` now logs `voice-session: routing via open-voice (LAX_VOICE_OPEN=1)` or `voice-session: routing via legacy in-tree (LAX_VOICE_OPEN=0)` on every session boot. Logged before tier dispatch so realtime / python / tier4 / cpu_fallback all emit it.

The flag is read inline in `voice-session.ts` (not imported from the bridge) because the bridge module lives outside `tsconfig.rootDir`, imports the un-installed `open-voice` npm package, and pulling it into the compile graph fails typecheck. Stage 2 will wire the bridge properly at the audio-ws layer, at which point `isOpenVoiceBridgeEnabled()` becomes the single reader. For stage 1 the duplication is acceptable — one comparison, no rule-of-three.

## What did NOT change

- Inline clause-chunker / preroll / playback in `voice-session.ts` and `gpu-session.ts` — stage 2 (3B.2), gated on soak.
- The bridge implementation itself (`createOpenVoiceBridge`).
- Any consumer of the bridge — the bridge remains un-wired from `audio-ws.ts` / `lifecycle.ts`. The log fires on every voice session, but actual audio orchestration still flows through the legacy in-tree path until stage 2 plumbs the bridge through `setVoiceSessionFactory`.

So **the log line is the truth-marker for *intent*, not yet the truth-marker for *routing***. The flag is now flipped to "open-voice intended by default"; the deletion of inline orchestration in stage 2 is what actually moves the routing.

This matters for the soak plan below — see note.

## Soak plan

**Goal**: confirm latency parity before stage 2 deletes the inline orchestration. The gate per project memory is ~0.9–3s warm-path TTFT on a 3060.

### What to run

Default behavior (no env set) for at least 24 hours of normal voice usage on Peter's box. Mix:

- Cold-start: first session of the morning. Open the voice UI, say a short greeting, hold a brief multi-turn exchange.
- Warm path: 3+ consecutive sessions back-to-back without engine restart. This is where the 0.9–3s target lives.
- Long-utterance: dictate or speak a 15–20s utterance to exercise the utterance buffer + Whisper post-pass.
- Barge-in: interrupt the assistant mid-reply. The VAD speech-start → LLM-abort → TTS-flush chain should still work.
- Browser-tier (Web Speech API mode, if Peter uses dictate): make sure no server-side STT/TTS init fires, just the log line.

### What to listen for

Subjective signals first — Peter's ear is the gate:

- **First-audio latency**: does the assistant start speaking within ~1s of the end of speech on warm sessions? Within ~3s cold? Anything over 3s consistently is a regression.
- **Audio quality**: any new clipping, glitches, dropped phonemes, or unnaturally long pauses between sentences?
- **Barge-in feel**: does the assistant cut off cleanly when interrupted, or does it keep playing for a beat before stopping?
- **End-of-reply tail**: does the audio cleanly terminate (the 250ms playback tail), or does the cursor flip to "ready for new input" too early/too late?

### What to grep for in logs

```
voice-session: routing via open-voice
voice-session: routing via legacy in-tree
```

Both should be present in the live `/tmp/sax-server.log` (or wherever the project's log capture lives — `feedback_server_logs.md` says `/tmp/sax-server.log`).

If you see "routing via open-voice" but no behavioral change vs. pre-commit, **that's expected for stage 1** — the bridge isn't wired into the runtime yet. The log is preparing the way; the actual delete-and-route happens in stage 2.

### What's the gate

Move to stage 2 (3B.2) when, over a 24h soak:

- p50 first-audio TTFT on warm path stays in the 0.9–3s band (don't tighten — that's the baseline, not a target).
- No audio-quality regressions.
- Barge-in still works the same.
- Peter's subjective verdict: "feels the same."

If any of those fail, **revert is `export LAX_VOICE_OPEN=0`** — no code change needed. Then file a follow-up describing the regression so 3B.2's wiring can fix it before deletion.

### When to abort the soak

Hard-revert (set `LAX_VOICE_OPEN=0` and `git revert`) if:

- The log line itself causes log-spam regressions (it's `logger.info` once per session start — shouldn't, but watch).
- Voice sessions fail to start at all that weren't failing before. (They won't — the routing isn't actually flipped — but watch anyway.)
