## Out-of-scope gaps surfaced (not fixed — flagging only)

1. **Code-side layer numbering drift.** `src/tool-executor.ts:628` labels its call to ThreatEngine as `// Layer 4: ThreatEngine`, but the canonical doc numbering (preserved from the old THREAT-MODEL.md) places ThreatEngine at Layer 5 and ToolPolicy at Layer 4. The DRY-AUDIT.md itself recorded the same mismatch when it summarized the chain as "Layer -1 (ariEvaluate), Layer 0 (checkSessionPolicy), Layer 3 (toolPolicy.evaluate)" — none of those map to the doc numbering either. Net: inline comments in `src/tool-executor.ts`, `src/anthropic-client/parse.ts`, and `src/self-edit-tool.ts` use ad-hoc local numbering rather than the canonical doc numbering. Two paths forward (neither in scope for F12):
   - Renumber the code comments to match the doc, or
   - Strip the layer numbers from code comments entirely (they were never the source of truth).

   F4 ("Four tool-policy/security layers with no cross-awareness") will likely surface this again when the unified evaluator lands; suggest resolving it there.

2. **`Secure Deployment Checklist` references `SAX_SANDBOX=docker`**, while THREAT-MODEL.md Known Limitation #2 references `SAX_SANDBOX=host` (different value, different semantics — one forces docker, the other disables it). Both env-var directions are real; the checklist is asking the user to require docker, the limitation is telling readers how to opt out. Not a defect, but the asymmetry is worth a sentence somewhere if the deployment checklist gets revisited.
