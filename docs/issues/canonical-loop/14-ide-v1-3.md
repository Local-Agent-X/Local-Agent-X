# Issue 14 — `ide` lane + session-pinned scheduling (v1.3)

**Phase:** v1.3 — session-pinned worker proof
**Status:** **BLOCKED** by v1.2 completion (build lane green, v1.2 tagged).
**Blocks:** Issue 15 (deletion gate)

---

## Goal

Bring the IDE flow onto canonical-loop by introducing the `ide` lane with `session_id`-pinned scheduling (cap=1 per session). Implements the existing IDE worker-session invariant — one active op per session at a time — through canonical-loop's lane mechanism rather than a bespoke orchestrator.

## Why it matters

The IDE flow is the last big special-case path. Once it runs on canonical-loop, all four lanes are migrated and the deletion gate becomes reachable. Session-pinning is also a unique scheduling shape (per-key cap, not just per-lane) and proves the scheduler can express it.

## Scope

- IDE adapter (likely a thin wrapper over the same Anthropic adapter, plus IDE-specific input prep). Reuse existing adapters where possible — no new adapter contract.
- `ide` lane:
  - Cap = 1 per `session_id`.
  - Different sessions can run concurrently up to a global ceiling (configurable, default reasonable e.g. 5).
  - FIFO within a session.
- Submission API: callers pass `session_id` for `lane='ide'`. Validation: `session_id` required for `ide` lane.
- Scheduler enforcement: never lease an op with `session_id=X` while another `session_id=X` op is `running`/`cancelling`.
- Behind flag `lax.canonical_loop.ide` (default OFF, opt-in canary).

## Hard constraint

**No contract changes.** No new lifecycle states. No new event types unless the canonical type set proves insufficient (which it should not).

## Non-goals

- Modifying existing IDE worker session flow until canonical proves itself in canary.
- Cross-session orchestration / fan-out.
- Multi-window per-session anything.
- New event types.

## Likely files / modules

- `src/canonical-loop/adapters/ide.ts` (if non-trivial) or reuse Anthropic adapter via configuration.
- `src/canonical-loop/scheduler.ts` — extend with per-`session_id` cap logic.
- `tests/canonical-loop/ide-lane.test.ts` — session-pinning scenarios.

## Dependencies / blockers

## Acceptance criteria

- `op_submit_async` for IDE lane requires `session_id`; missing `session_id` returns documented error.
- Two ops with same `session_id` cannot run concurrently — second waits for first to terminate.
- Two ops with different `session_id` can run concurrently up to global IDE cap.
- Conformance suite A–I passes for the IDE-bound adapter.
- Real IDE smoke: an IDE-style multi-turn op completes end-to-end through canonical-loop.
- No regression in v1.0 / v1.1 / v1.2.
- No diff in existing IDE worker session flow.

## Tests required

- Same-session serialization: submit 2 ops with same session_id → second blocks until first terminal.
- Different-session concurrency: submit 3 ops across 3 sessions → all three run.
- Cancel one session does not affect another.
- Crash recovery on IDE lane.
- Real IDE smoke (gated).
